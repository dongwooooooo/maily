## 제품 경계

Maily는 여러 Gmail 계정에서 중요한 메일을 선별해 브리핑하고, 사용자가 지금 처리할 일과 나중에 봐도 되는 일을 구분하도록 돕는다.

Gmail은 계속 원본 메일 시스템이다. Maily는 Gmail 위의 브리핑, 우선순위, 정리 보조 레이어로 동작한다.

## 포함 범위

- Google 로그인과 서비스 워크스페이스
- 여러 Gmail 계정 연결
- 연결된 Gmail 계정의 지속 동기화
- 전체 계정/계정별 오늘 브리핑
- 메일 요약, Gmail 상태 표시, 동기화 상태 표시
- Gmail에서 열기, 읽음 처리, 라벨 적용, 아카이브
- 라벨/아카이브 제안과 적용 상태
- 정리 제안 승인 큐
- 활동 로그와 Undo
- 브리핑 섹션/라벨 사용자화
- 알림 클릭 시 올바른 맥락으로 진입

## 제외 범위

- 자체 메일 작성, 답장, 발송
- Gmail 전체 대체 inbox
- 모바일 앱
- 긴 온보딩 튜토리얼
- 상세한 AI 판단 이유 기본 노출
- 승인 없는 대량 정리 확정

## 기능별 설명

### F01. 서비스 로그인/워크스페이스

사용자는 대표 Google 계정으로 Maily에 로그인한다. 이 계정은 서비스 사용자를 식별하는 계정이며, 브리핑 대상 Gmail 계정과 반드시 같을 필요는 없다.

재로그인 시 기존 워크스페이스로 돌아와야 하며, 같은 사용자가 로그인할 때마다 새 작업공간이 만들어지면 안 된다.

### F02. Gmail 계정 연결

사용자는 설정에서 Gmail 계정을 추가한다. 연결 계정은 메일 소스이며, 상단 계정 스코프와 설정 화면에서만 관리한다.

계정별로 표시 이름, 브리핑 포함 여부, AI 요약 사용 여부, 알림 사용 여부, 일시정지를 설정할 수 있다. 권한이 끊기거나 토큰 갱신이 실패하면 해당 계정은 `permission_needed` 상태로 보여야 한다.

### F03. 지속 Gmail 동기화

Maily는 사용자가 매번 수동 동기화를 누르지 않아도 연결된 Gmail 계정의 새 메일과 상태 변화를 서비스에 반영해야 한다.

목표 동작은 Gmail 계정별 변경 알림을 받고, 마지막으로 처리한 Gmail history cursor 이후의 변경분만 가져와 message snapshot과 브리핑 후보를 갱신하는 것이다. 알림 누락, watch 만료, cursor 오류가 생길 수 있으므로 정기적인 fallback 동기화와 full resync 복구가 필요하다.

이 기능은 프론트 디자인과 병렬로 가장 먼저 POC를 돌려야 하는 백엔드 리스크다.

### F04. 오늘 브리핑

첫 화면은 전체 inbox가 아니라 오늘 브리핑이다. 사용자는 전체 계정 또는 특정 Gmail 계정 범위에서 중요한 항목, 답장 필요, 나중에 봐도 됨, 정리됨, 승인 필요 항목을 훑는다.

메일 카드는 스캔과 선택만 담당한다. 카드 안에는 액션 버튼, 판단 이유, New 배지, 컬러 바를 넣지 않는다.

### F05. 메일 상세

사용자가 카드를 선택하면 상세 패널에서 Gmail 원문 링크, 요약, 제한된 본문 발췌, 현재 Gmail 상태, 비서 처리 결과를 확인한다.

상세 패널은 Gmail 변경 액션을 실행할 수 있는 첫 번째 위치다. 목록 카드에서 바로 Gmail을 바꾸지 않는다.

### F06. Gmail 변경 액션

사용자는 상세 패널 또는 정리 검토에서 Gmail 읽음 처리, 라벨 적용, 아카이브, 읽음 처리 후 아카이브를 실행할 수 있다.

Gmail에 실제 변경이 생기면 응답과 UI에는 변경 결과, 활동 로그, Undo 가능 여부가 명확히 남아야 한다. Gmail 변경이 없으면 `브리핑만 생성됨`처럼 변경 없음도 명시한다.

### F07. 라벨/분류

기본 브리핑 섹션은 상태에서 파생되는 목록이다. 사용자가 직접 이동시키는 목적지는 기본 섹션이 아니라 사용자 라벨이다.

사용자 라벨은 Gmail `Maily/` 라벨과 동기화된다. 라벨 이름 변경, 숨김, 순서 변경은 서비스 라벨과 Gmail 라벨 매핑을 깨지 않아야 한다.

### F08. 다음부터 여기로

사용자가 메일을 특정 라벨로 이동하거나 분류를 고치면, Maily는 유사한 메일을 다음부터 같은 방식으로 처리할 규칙 후보를 만들 수 있다.

규칙은 자동으로 활성화하지 않는다. 사용자가 평문으로 확인하고 승인한 뒤 적용한다.

### F09. 보관함

보관함은 지난 브리핑을 쌓아두는 곳이 아니다. 보관함은 앞으로 볼 항목의 예정 타임라인과 사용자 라벨 허브를 담당한다.

`나중에` 상태의 항목은 지정 시각이나 조건이 되었을 때 다시 브리핑/알림 후보가 된다.

### F10. 정리 검토

Maily가 낮은 확신으로 판단한 정리 제안은 정리 검토 큐에 들어간다. 사용자는 제안을 한 항목씩 승인하거나 제외한다.

일괄 승인 버튼은 두지 않는다. 제안 기준에 못 미치는 항목은 큐에 보여주지 않는다.

### F11. 활동 로그/Undo

활동 로그는 자동 처리와 수동 Gmail 변경을 모두 기록한다. 신뢰는 숨겨진 자동화가 아니라 보이는 변경 이력에서 나온다.

Undo는 가능한 액션에서만 제공한다. 모든 Gmail 변경이 되돌릴 수 있다고 약속하지 않는다.

### F12. 요약/개인정보

AI 요약은 계정별로 켜고 끌 수 있다. 요약이 꺼진 계정은 메타데이터 카드와 Gmail 열기만 제공한다.

요약에는 필요한 최소 내용만 사용한다. 원시 프롬프트와 전체 원문을 장기 보관하지 않는다.

### F13. 알림/라우팅

알림은 일반 landing page가 아니라 기존 화면과 선택 항목으로 연결된다.

예: 중요 메일 알림은 오늘 브리핑의 해당 항목, 재알림은 상세 패널, 정리 승인은 정리 검토, 권한 오류는 연결 계정 설정으로 이동한다.

### F14. 동기화/오류 복구

동기화 상태는 상단 계정 컨트롤 안에서 컴팩트하게 보여준다. 첫 연결 순간을 제외하고 큰 진행 화면으로 사용자를 막지 않는다.

권한 필요, 오류, 일시정지는 계정별 상태로 보여주며, 사용자가 어떤 계정에 조치해야 하는지 알 수 있어야 한다.

## MVP 1차 범위

1. 가짜 데이터 기반 프론트엔드 셸
2. 오늘 브리핑, 메일 카드, 우측 상세 패널, 상단 계정 스코프, 사이드바, Undo toast
3. 로그인, 첫 Gmail 적용 확인, 빈 상태, 상태 변형, 보관함, 알림 진입, 설정, 정리 검토, 활동 로그의 정적 화면
4. 계정, 섹션, 메일, 라벨, 정리 제안, 알림, 활동 로그 mock 데이터 모델
5. 선택 상태, 상세 액션, toast/Undo, 섹션 앵커, 간단한 화면 전환

## 백엔드 우선 POC 범위

프론트 디자인과 병렬로 아래 백엔드 기능은 먼저 POC/TDD로 검증한다.

1. F01-F02: 서비스 로그인 계정과 연결 Gmail 계정의 분리
2. F03: Gmail 지속 동기화, watch 갱신, history cursor, fallback sync
3. F04-F05: 프론트 mock을 대체할 브리핑/상세 읽기 API
4. F06/F11: Gmail 변경 액션과 활동 로그/Undo 계약
5. F07/F10: 라벨/규칙/정리 검토 계약

## 완료 판정

- 전체 inbox 없이 오늘 중요한 내용을 이해할 수 있다.
- 모든 항목에서 Gmail 계정 맥락이 보인다.
- 새로 도착한 Gmail 메일이 수동 새로고침 없이 브리핑 후보에 반영된다.
- Gmail 변경 여부와 변경 결과가 명확하다.
- Gmail을 열기 전에 필요한 처리 판단을 할 수 있다.
- 사용자 라벨과 향후 분류 규칙이 표현된다.
- 동기화, 권한, 알림 상태가 메인을 방해하지 않는다.
- 알림 클릭이 특정 맥락으로 착지한다.
- 설정에서 계정과 AI 요약 옵션을 관리할 수 있다.
